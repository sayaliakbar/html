Introduction to Accessibility

Web accessibility means that websites, tools, and technologies are designed and developed so that people with disabilities can use them. More specifically, people can:

  • Perceive, understand, navigate, and interact with the Web
  • Contribute to the Web

Web accessibility encompasses all disabilities that affect access to the Web, including:

  • Visual: Blindness, low vision, color-blindness
  • Auditory: Deafness and hearing loss
  • Motor: Limited fine motor control, slower response time, limited movement
  • Cognitive: Learning disabilities, distractibility, inability to focus on large amounts of information

Why Accessibility Matters:

  • Ethical Consideration: The web is designed as an universally accessible medium
  • Legal Requirement: Many countries have laws requiring digital accessibility
  • Business Value: Accessible sites reach more users and improve SEO
  • Universal Design: Accessible design benefits all users, not just those with disabilities

WCAG Guidelines

The Web Content Accessibility Guidelines (WCAG) are developed by the W3C and provide a framework for making web content more accessible. WCAG is organized around four principles, often referred to as POUR:

1. Perceivable

Information and user interface components must be presentable to users in ways they can perceive.

  • Provide text alternatives for non-text content
  • Provide captions and alternatives for multimedia
  • Create content that can be presented in different ways
  • Make it easier for users to see and hear content

2. Operable

User interface components and navigation must be operable.

  • Make all functionality available from a keyboard
  • Give users enough time to read and use content
  • Do not use content that causes seizures or physical reactions
  • Help users navigate and find content

3. Understandable

Information and the operation of the user interface must be understandable.

  • Make text readable and understandable
  • Make content appear and operate in predictable ways
  • Help users avoid and correct mistakes

4. Robust

Content must be robust enough to be interpreted by a wide variety of user agents, including assistive technologies.

  • Maximize compatibility with current and future user tools
  • Use valid, well-formed markup

Conformance Levels

WCAG has three levels of conformance:

  • Level A: Minimum level of accessibility
  • Level AA: Addresses the most common barriers for disabled users (most common target for legal compliance)
  • Level AAA: Highest level of accessibility

Semantic HTML

Using semantic HTML elements is one of the most fundamental ways to improve accessibility. Semantic elements provide meaning and structure to the content, which helps assistive technologies understand and navigate it.

Key Semantic Elements

Element Purpose
<header> Introductory content, typically containing headings, logos, navigation
<nav> Section with navigation links
<main> Main content of the document
<article> Self-contained composition (e.g., blog post, forum post)
<section> Standalone section of a document
<aside> Content tangentially related to the main content
<footer> Footer for the page or section
<figure> and <figcaption> Self-contained content referenced in the main flow with optional caption

Bad Example:

<div class="header">
  <div class="logo">My Site</div>
  <div class="nav">
    <div class="nav-item"><a href="#">Home</a></div>
    <div class="nav-item"><a href="#">About</a></div>
  </div>
</div>

<div class="main">
  <div class="content">
    <div class="title">Welcome to My Site</div>
    <div class="text">This is a sample website.</div>
  </div>
  
  <div class="sidebar">
    <div class="widget">Recent Posts</div>
  </div>
</div>

<div class="footer">
  <div>Copyright 2023</div>
</div>

Good Example:

<header>
  <h1>My Site</h1>
  <nav>
    <ul>
      <li><a href="#">Home</a></li>
      <li><a href="#">About</a></li>
    </ul>
  </nav>
</header>

<main>
  <article>
    <h2>Welcome to My Site</h2>
    <p>This is a sample website.</p>
  </article>
  
  <aside>
    <h3>Recent Posts</h3>
  </aside>
</main>

<footer>
  <p>Copyright 2023</p>
</footer>

Proper Heading Structure

Headings (<h1> through <h6>) provide hierarchical structure to your content. Screen reader users often navigate by headings.

Bad Example:

<div class="title">My Website</div>
<div class="subtitle">Welcome</div>
<div class="section-title">About Us</div>

Good Example:

<h1>My Website</h1>
<h2>Welcome</h2>
<h2>About Us</h2>

WAI-ARIA

WAI-ARIA (Web Accessibility Initiative - Accessible Rich Internet Applications) helps make dynamic content and advanced user interface controls developed with HTML, JavaScript, and related technologies more accessible.

Important: Only use ARIA when necessary. The first rule of ARIA is: if you can use a native HTML element or attribute with the semantics and behavior you require, then do so.

Key ARIA Concepts

Roles

Roles define what an element is or does. They communicate the purpose of an element to assistive technologies. 

<div role="button" tabindex="0">Submit</div>

This div will be announced as a button to screen readers. (But better to just use <button> when possible).

States and Properties

These provide specific information about an object, or describe relationships and associations between objects. 

<button aria-expanded="false" aria-controls="menu1">Menu</button>
<ul id="menu1" hidden>
  <li><a href="#">Option 1</a></li>
  <li><a href="#">Option 2</a></li>
</ul>

This code tells screen readers that the button controls a menu that is currently collapsed.

Common ARIA Attributes

Attribute Purpose
aria-label Provides an accessible name when visible text isn't available
aria-labelledby References another element to serve as the accessible name
aria-describedby References another element that provides additional description
aria-expanded Indicates if a control is expanded or collapsed
aria-hidden Indicates if an element should be hidden from assistive technology
aria-live Indicates that an element will be updated dynamically
aria-required Indicates that user input is required
aria-invalid Indicates that input is invalid

Live Regions

Live regions allow you to inform screen reader users of dynamic content changes. 

<div aria-live="polite" id="notification">
  Your form has been submitted successfully.
</div>

When content changes within this div, screen readers will announce it without shifting focus.

Common values for aria-live:

  • polite: Announces changes when the user is idle
  • assertive: Announces changes immediately (use sparingly)
  • off: Does not announce changes

Keyboard Accessibility

Many users with motor disabilities, as well as screen reader users, rely on keyboard navigation. All functionality should be available using only a keyboard.

Focus Management

Managing focus is crucial for keyboard accessibility. Users should always know where they are on the page.

Making Elements Focusable

Interactive elements should be naturally focusable. For custom controls, use tabindex:

  • tabindex="0": Includes the element in natural tab order
  • tabindex="-1": Makes the element focusable by script but not by tab navigation
  • tabindex="1+": Gives the element priority in tab order (generally avoid this) 
<div tabindex="0" role="button" id="customButton">Click Me</div>

<script>
document.getElementById('customButton').addEventListener('keydown', function(e) {
  if (e.key === 'Enter' || e.key === ' ') {
    // Handle activation
    e.preventDefault();
    this.click();
  }
});
</script>

Skip Links

Skip links allow keyboard users to bypass repetitive navigation menus and jump directly to the main content. 

<a href="#main" class="skip-link">Skip to main content</a>

<nav>
  <!-- Navigation items -->
</nav>

<main id="main" tabindex="-1">
  <!-- Main content -->
</main>

CSS for the skip link:

.skip-link {
  position: absolute;
  top: -40px;
  left: 0;
  background: #2c3e50;
  color: white;
  padding: 8px;
  z-index: 100;
}

.skip-link:focus {
  top: 0;
}

Focus Indication

Users should be able to see which element is currently focused. Never completely remove focus indicators with outline: none without providing an alternative.

Good Focus Style:

:focus {
  outline: 3px solid #2196f3;
  outline-offset: 2px;
}

/* If removing outline, provide an alternative */
button:focus {
  outline: none;
  box-shadow: 0 0 0 3px #2196f3;
}

Visual Accessibility

Color Contrast

Text should have sufficient contrast with its background to be readable by people with low vision or color blindness.

  • WCAG AA requires a contrast ratio of at least 4.5:1 for normal text and 3:1 for large text
  • WCAG AAA requires a contrast ratio of at least 7:1 for normal text and 4.5:1 for large text

Use tools like the WebAIM Contrast Checker to verify your color combinations.

Color Independence

Don't rely solely on color to convey information. Always include additional indicators such as text, icons, or patterns.

Bad Example:

<p>Fields marked in red are required.</p>
<input type="text" style="border-color: red;">

Good Example:

<p>Fields marked with * are required.</p>
<label>Name *
  <input type="text" aria-required="true">
</label>

Resizable Text

Text should be resizable up to 200% without loss of content or functionality. Use relative units like em, rem, and percentages rather than fixed pixel sizes.

Images and Media

All images should have alternative text that describes the information or function of the image. 

<img src="logo.png" alt="Company Name">

<!-- For decorative images -->
<img src="decorative-line.png" alt="">

<!-- For complex images -->
<figure>
  <img src="chart.png" alt="Bar chart showing sales growth from 2020-2023">
  <figcaption>Figure 1: Sales have increased by 25% year over year</figcaption>
</figure>

Multimedia

Video and audio content should include alternatives:

  • Videos should have captions for deaf and hard of hearing users
  • Audio content should have transcripts
  • Consider adding audio descriptions for videos when important visual information is not covered in the dialogue 
<video controls>
  <source src="video.mp4" type="video/mp4">
  <track src="captions.vtt" kind="subtitles" srclang="en" label="English">
  Your browser does not support the video tag.
</video>

Testing & Validation

Automated Testing Tools

Automated tools can help identify many accessibility issues, but they can only catch about 30-40% of all problems.

  • Lighthouse: Built into Chrome DevTools
  • WAVE: Web Accessibility Evaluation Tool by WebAIM
  • axe: Accessibility testing engine
  • HTML Validators: Ensure valid, well-formed markup

Manual Testing

Manual testing is essential to catch issues that automated tools miss:

  • Keyboard navigation: Try using your site with only a keyboard
  • Screen reader testing: Use VoiceOver (Mac), NVDA or JAWS (Windows), or TalkBack (Android)
  • Color contrast: Check text readability at different sizes
  • Zoom testing: Test with page zoomed to 200%
  • Reduced motion: Test with reduced motion settings enabled

Accessibility Checklist

A basic checklist for testing your pages:

  1. Page has a proper heading structure (h1 - h6)
  2. Images have appropriate alt text
  3. Color contrast meets WCAG AA guidelines
  4. Interactive elements can be accessed via keyboard
  5. Focus is visible on all interactive elements
  6. Page is navigable with a screen reader
  7. Forms have proper labels and error messages
  8. Dynamic content changes are announced to screen readers
  9. Page structure uses semantic HTML elements
  10. ARIA is used appropriately when HTML isn't sufficient
